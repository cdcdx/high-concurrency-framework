# 高并发业务系统需求文档

---

## 一、系统定位与背景

### 1.1 业务定位
面向 C 端用户的通用高并发业务中台，承载核心交易链路（订单）、用户服务、数据分析等场景。系统需在流量洪峰下保持稳定，提供秒级/亚秒级响应体验。

### 1.2 规模目标
- 总用户量：**千万级**
- 在线用户：**百万级**
- 峰值 QPS（读）：**50000+**
- 峰值 TPS（写）：**5000+**
- 日订单量：**百万级**
- 数据总量：**TB 级**

### 1.3 核心挑战
- 热点数据瞬时高并发读（如热门商品库存、爆款订单查询）
- 写流量洪峰冲击核心交易库（如秒杀下单）
- 多数据源异构带来的一致性与延迟权衡
- 大规模分布式部署下的故障隔离与优雅降级

---

## 二、设计原则

| 原则 | 说明 |
|------|------|
| **无状态优先** | 业务 Pod 不存本地状态，所有状态外置至 Redis/DB/Kafka，确保任意 Pod 可互换 |
| **防御性降级** | 任意外部依赖故障时，相关功能自动降级而非整体崩溃，服务保持部分可用 |
| **就近缓存** | 热数据尽可能命中 L1 本地缓存（<1ms），逐级回源，减少网络 IO |
| **异步削峰** | 写操作超阈值时自动切换 Kafka 异步通路，保证前端低延迟响应（<10ms） |
| **读写分离** | 核心数据库 Master 写 / Replica 读，分析型查询路由至专用 OLAP 引擎 |
| **可观测先行** | 全链路 TraceID + 结构化日志 + 内置 Metrics 端点，故障可快速定位 |
| **水平可扩展** | 每层均可独立横向扩容，无单点瓶颈 |

## 三、系统架构

### 3.1 请求链路
```
客户端
  │
  ▼
L7 负载均衡 (Nginx/K8s Ingress)
  │
  ▼
API Gateway / 服务层 (Go/Gin)
  │
  ├── 读路径 ──→ L1 本地缓存 (Ristretto, <1ms)
  │               │ (miss)
  │               ▼
  │              L2 分布式缓存 (Redis Cluster, 1-5ms)
  │               │ (miss)
  │               ▼
  │              L3 持久化层 (MySQL/PG/MongoDB, 10-200ms)
  │
  └── 写路径 ──→ 令牌桶限流器 (5000 TPS)
                  │
          ┌───────┴───────┐
          ▼               ▼
     同步通路          异步通路
   (直写 MySQL)    (Kafka → Consumer → MySQL)
   + 分布式锁          + 死信队列(DLQ)
   + 延迟双删          + 速率控制(2000/s)
          │               │
          └───────┬───────┘
                  ▼
           ES 异步索引 / PG 行为日志
```

### 3.2 关键决策点

**写操作分流判定逻辑：**
```
if 熔断器 == OPEN:
    → 强制走异步通路（降级）
elif 令牌桶.TryAcquire() == true:
    → 尝试获取分布式锁
       ├── 成功 → 同步直写 MySQL + 延迟双删缓存
       └── 失败 → 降级走异步通路
else:
    → 异步通路（Kafka 削峰）
```

**缓存查询回源逻辑：**
```
L1.Get(key)
  ├── hit  → 直接返回（<1ms）
  └── miss → L2.Get(key)
                ├── hit  → 回填 L1，返回（1-5ms）
                └── miss → 热点Key检测
                              ├── 是热点 → Redis SETNX 互斥锁
                              │             ├── 获取锁 → 查 L3 DB，回填 L1+L2，释放锁
                              │             └── 未获取 → 等待 + 重试
                              └── 非热点 → 查 L3 DB，回填 L1+L2
                                           └── DB 返回 nil → 缓存空标记（60s 防穿透）
```

## 四、功能需求

### 4.1 用户认证模块

| ID | 功能 | 接口 | 优先级 |
|----|------|------|--------|
| AUTH-01 | 用户注册（用户名/密码/邮箱/手机号，bcrypt 哈希） | `POST /api/v1/auth/register` | P0 |
| AUTH-02 | 用户登录（用户名+密码，返回 JWT Token，2h 有效） | `POST /api/v1/auth/login` | P0 |
| AUTH-03 | 获取当前用户信息（从 JWT 解析身份） | `GET /api/v1/auth/me` | P1 |
| AUTH-04 | JWT 鉴权中间件（所有 `/api/v1/*` 除 auth 外均校验） | 中间件 | P0 |
| AUTH-05 | 账号禁用/启用的状态管理（status 字段） | 登录时校验 | P1 |
| AUTH-06 | 登录时间异步更新（不阻塞登录响应） | — | P2 |

- 用户名 3-64 字符，密码 6-128 字符
- 用户名/邮箱全局唯一，重复返回 409

## 六、韧性体系（故障容错）

### 6.1 三级缓存防护矩阵

| 风险 | 防护策略 | 具体实现 |
|------|----------|----------|
| **缓存穿透**（查询不存在的数据） | 空值缓存 | DB 返回 nil 时缓存 `__NULL__` 标记，TTL=60s |
| **缓存击穿**（热点 Key 过期瞬间高并发） | 互斥锁回源 | 热点 Key 检测 + Redis SETNX 分布式锁，串行回源 |
| **缓存雪崩**（大量 Key 同时过期） | TTL 随机偏移 | 所有 TTL ±20% 随机抖动（Jitter），防止集中过期 |
| **数据不一致**（写后缓存脏数据） | 延迟双删 | 写后立即删缓存 + 500ms 后二次删除 |
| **跨 Pod 缓存不一致** | 失效广播 | L2 变更发布 Pub/Sub 事件，各 Pod 订阅并清理本地 L1 |

### 6.2 热点 Key 检测

- **算法：** 滑动窗口，10 个时间槽，窗口大小 10s（可配置）
- **判定阈值：** 窗口内访问次数 > 100（可配置）
- **生命周期：** 检测为热点 → 加分布式锁回源 → 定期清理低频 Key

### 6.3 熔断器（Circuit Breaker）

- **三态模型：** CLOSED → OPEN → HALF_OPEN → CLOSED
- **触发条件：** 统计窗口（60s）内失败率 ≥ 50%（可配置）
- **慢调用阈值：** 单次调用 > 1000ms（可配置）
- **半开探测：** OPEN 30s 后转入 HALF_OPEN，最多放行 5 个请求验证恢复
- **熔断时行为：** 写操作强制全量走异步降级，读操作可配置
- **后台维护：** 两个协程分别负责滑动窗口重置 + 状态转换检测

### 6.4 令牌桶限流

| 参数 | 值 | 说明 |
|------|-----|------|
| 桶容量 | 5000 | 最大突发流量 |
| 补充速率 | 5000/s | 稳态吞吐 |
| TryAcquire | 非阻塞 | 用于分流判定，不阻塞请求 |
| WaitAcquire | 阻塞 | 支持 Context 超时取消 |

### 6.5 分布式锁

- **实现：** Redis SETNX + 过期时间（防死锁）
- **解锁：** Lua 脚本原子操作（校验 token 匹配 → 删除）
- **锁令牌：** `{纳秒时间戳}-{8位随机Hex}`，唯一标识
- **高阶封装：** `WithLock` 函数，锁获取失败自动降级

### 6.6 Kafka 消息可靠性

| 层级 | 保障 |
|------|------|
| 生产者 | acks=all（所有 ISR 确认） + 幂等 + LZ4 压缩 + 最多 3 次重试 |
| 消费者 | 手动提交 Offset + 指数退避重试（最多 3 次） + 死信队列（DLQ） |
| 补偿 | 发送失败时写入本地补偿日志文件（可供 FileBeat 采集重放） |
| 追踪 | TraceID 注入 Kafka Headers，消费者还原到 Context |

### 6.7 组件级降级策略

| 依赖不可用 | 降级行为 | 影响范围 |
|------------|----------|----------|
| Redis 不可用 | L1 直接跳过 L2 查 L3 数据库 | 读延迟上升至 10-200ms |
| Kafka 不可用 | 异步消息写入内存 Mock（测试）/ 降级日志（生产） | 异步通路中断，同步不受影响 |
| MySQL Master 不可用 | 读可走 Replica（如有），写操作降级 | 写功能受限 |
| PostgreSQL 不可用 | 分析接口返回降级响应 | 分析功能不可用 |
| MongoDB 不可用 | 用户资料接口降级 | 用户资料功能不可用 |
| Elasticsearch 不可用 | 搜索接口返回 503 | 全文搜索不可用 |

## 七、多数据库策略

### 7.1 存储选型依据

| 数据库 | 场景 | 选型理由 | 替代方案 |
|--------|------|----------|----------|
| **MySQL** | 订单 + 用户认证 | OLTP 场景，ACID 事务，行级锁，成熟稳定 | —（核心交易不可替代） |
| **PostgreSQL** | 日度统计 + 行为日志 | OLAP 场景，窗口函数/CTE/JSONB，分析能力强 | 简单聚合场景可用 MySQL 替代，引入 PG 增加运维复杂度；仅在分析需求复杂（多表 JOIN、时序聚合、地理空间）时性价比成立 |
| **MongoDB** | 用户资料 | 非结构化数据，Schema 灵活，嵌套文档，upsert 天然支持 | MySQL JSON 列可替代简单场景，但嵌套文档深度大时 MongoDB 优势明显 |
| **Elasticsearch** | 订单/用户全文搜索 | 倒排索引，分词搜索，多字段权重匹配，近实时 | MySQL FULLTEXT / PostgreSQL tsvector 可替代简单搜索，但中文分词、权重调优时 ES 更专业 |

**选型权衡原则：**
- 初创期（数据 < 100GB，分析需求简单）：MySQL 单一 RDBMS 即可覆盖 OLTP + 简单 OLAP，降低运维成本
- 成长期（数据分析需求增长）：引入 PostgreSQL 做报表/BI 查询，与 OLTP 物理隔离
- 成熟期（TB 级数据，多维分析）：PG + ClickHouse 组合，ES 做日志分析
- 当前框架为**示范架构**，保留 PostgreSQL 以展示 OLTP/OLAP 分离的最佳实践

### 7.2 连接池配置

| 数据库 | 最大连接 | 最小空闲 | 最大生存期 | 备注 |
|--------|----------|----------|------------|------|
| MySQL | 50 | 10 | 1800s | 支持读写分离，Replica 轮询 |
| PostgreSQL | 30 | 5 | 1800s | 支持读写分离 |
| MongoDB | 100 | 10 | — | 支持 ReadPreference 配置 |
| Elasticsearch | — | — | — | 异步索引，不维护连接池 |

### 7.3 数据一致性边界

- **MySQL → ES：** 最终一致性（异步索引，秒级延迟），允许短暂不一致
- **MySQL → PG 行为日志：** 最终一致性（异步写入），允许丢失（丢弃 > 丢弃）
- **缓存 → 数据库：** 延迟双删保证最终一致性，允许毫秒级脏读窗口
- **Kafka → DB：** at-least-once（消费端去重依赖幂等设计）

## 九、技术风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| Redis 集群分片不均 | 热点分片过载打垮部分节点 | 热点 Key 检测 + L1 本地缓存吸收 |
| Kafka 消费积压 | 异步订单延迟入库 | 消费速率可配（2000/s），增加消费者并发度（8 协程） |
| 数据库主从延迟 | 读 Replica 拿到旧数据 | 关键读走 Master，缓存层拦截大部分请求 |
| 缓存与 DB 数据不一致 | 返回过期/脏数据 | 延迟双删 + TTL 兜底 |
| Pod 频繁扩缩（抖动） | 服务不稳定 | HPA 稳定窗口（30s up / 5min down） |
| JWT Token 泄露 | 用户身份被冒用 | 短期有效（2h）+ 生产强制修改 Secret |
| 内存泄漏 | Pod OOM Kill | L1 缓存设置 max_entries 硬上限，Goroutine 限流 |
