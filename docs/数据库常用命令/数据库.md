# 数据库/数据存储 全量汇总与分析

> 覆盖市面主流数据库与数据存储系统，按类别分组，分析同类产品的差异、适用场景、核心特点、性能参数、横向扩展能力。

---

## 一、关系型数据库 (RDBMS)

| 数据库 | 水平扩展 | 一致性 | 读 QPS (单机) | 写 QPS (单机) | P99 延迟 | 适用规模 |
|--------|:--:|--------|:--:|:--:|:--:|:--:|
| MySQL | ❌ 困难 | ACID | 1万~5万 | 5千~2万 | 1~10ms | 中等 |
| PostgreSQL | ❌ 困难 | ACID | 1万~5万 | 5千~2万 | 1~10ms | 中等 |
| MariaDB | ❌ 困难 | ACID | 1万~5万 | 5千~2万 | 1~10ms | 中等 |
| SQLite | ❌ | ACID | 1千~1万 | ~100 (串行) | 1~5ms | 极小 |
| TiDB | ✅ 弹性扩容 | 分布式 ACID | 10万~100万(集群) | 5万~50万(集群) | 5~50ms | 超大 |
| CockroachDB | ✅ 弹性扩容 | 分布式 ACID | 5万~50万(集群) | 3万~30万(集群) | 10~100ms | 超大 |
| OceanBase | ✅ 弹性扩容 | 分布式 ACID | 10万~100万(集群) | 5万~80万(集群) | 1~5ms | 超大 |

### 1. MySQL
- **数据类型**：结构化表数据
- **典型场景**：用户信息、订单、商品、账户、电商系统、CMS、ERP 等
- **核心特点**：ACID 事务、强一致性、成熟生态、SQL 标准、适合复杂关联查询（JOIN）
- **缺点**：单表过亿性能骤降；水平扩展需应用层介入（ShardingSphere/Vitess）；JSON 查询能力弱
- **性能参数**：
  - 响应延迟：P50 < 1ms, P99 1~10ms（单机，主键查询）
  - 读 QPS：1万~5万（单机，InnoDB 高并发读）
  - 写 QPS：5千~2万（单机，InnoDB 批量写入）
  - 备注：读写分离可提升读 QPS 3~10 倍；分库分表后可线性扩展

### 2. PostgreSQL (psql)
- **数据类型**：结构化 + JSON/空间/时序
- **典型场景**：GIS 地理空间数据、JSON 文档存储、复杂分析查询、全文搜索
- **核心特点**：扩展性极强，原生支持数组、JSONB、PostGIS、自定义类型，可做 OLAP+OLTP 混合
- **缺点**：原生水平扩展弱；大量并发写入不如 Cassandra；Vacuum 机制需额外关注
- **性能参数**：
  - 响应延迟：P50 < 1ms, P99 1~10ms（单机，简单查询）
  - 读 QPS：1万~5万（单机，索引命中）
  - 写 QPS：5千~2万（单机，包含索引更新）
  - 备注：复杂分析查询延迟较高（秒级）；Citus 插件可扩展读/写

### 3. MariaDB
- **数据类型**：结构化表数据（MySQL 分支）
- **典型场景**：MySQL 替代方案、需要更开放的存储引擎（MyRocks/ColumnStore）的场景
- **核心特点**：MySQL 协议兼容、Galera 集群支持好、内置更多存储引擎、发行版更活跃
- **缺点**：生态工具略少于 MySQL；部分 MySQL 特性迁移需注意兼容性差异
- **性能参数**：
  - 响应延迟：P50 < 1ms, P99 1~10ms（与 MySQL 同级）
  - 读 QPS：1万~5万（单机）
  - 写 QPS：5千~2万（单机，Galera 同步复制会略降写性能）
  - 备注：性能与 MySQL 基本持平；Galera 多主模式下写延迟受网络 RTT 影响

### 4. SQLite
- **数据类型**：本地轻量结构化数据
- **典型场景**：移动 App（iOS/Android）、桌面应用、嵌入式设备、边缘计算、开发测试
- **核心特点**：零配置、无服务端进程、单文件存储、ACID 事务
- **缺点**：不支持网络访问、不支持高并发写入、无水平扩展
- **性能参数**：
  - 响应延迟：P50 < 0.5ms（内存）, < 5ms（磁盘）
  - 读 QPS：1千~5万（单文件，取决于缓存命中率）
  - 写 QPS：~100（默认同步模式）, 可达 5万+（WAL 模式 + 批量事务）
  - 备注：写入为串行（单写锁），高并发写场景不适合

### 5. TiDB
- **数据类型**：分布式关系型结构化数据
- **典型场景**：海量 OLTP 业务（订单/支付）、HTAP 混合负载、零停机弹性伸缩
- **核心特点**：MySQL 协议兼容、计算存储分离、自动分区（Raft）、自带 HTAP 能力
- **缺点**：运维复杂度高；延迟略高于单机 MySQL；复杂事务性能弱于单机数据库
- **性能参数**：
  - 响应延迟：P50 1~5ms, P99 5~50ms（3 副本，OLTP 点查）
  - 读 QPS：10万~100万+（水平扩展，随节点数线性增长）
  - 写 QPS：5万~50万+（水平扩展，Raft 共识有额外开销）
  - 备注：对比单机 MySQL，P99 延迟通常高 2~5 倍；TiFlash 列存可承担分析任务

### 6. CockroachDB
- **数据类型**：分布式关系型（PG 兼容）
- **典型场景**：跨地域多活业务、全球化部署、需要 PostgreSQL 兼容的弹性数据库
- **核心特点**：PG 协议兼容、天然多 Region 部署、Raft 副本、强一致性
- **缺点**：社区小于 TiDB；国内案例少；SQL 兼容性不如 TiDB 对接 MySQL 生态
- **性能参数**：
  - 响应延迟：P50 2~10ms, P99 10~100ms（跨 Region 延迟更高）
  - 读 QPS：5万~50万+（集群，受地理距离影响）
  - 写 QPS：3万~30万+（集群，Raft 多 Region 同步写入）
  - 备注：跨地域部署时延迟由网络 RTT 决定；不适合低延迟高频事务

### 7. OceanBase
- **数据类型**：金融级分布式关系型
- **典型场景**：金融核心（支付/账户）、对数据强一致性和高可用要求极高的企业级场景
- **核心特点**：Paxos 共识、准内存 OLTP + OLAP 混合、MySQL 协议兼容、蚂蚁集团出品
- **缺点**：社区版功能受限；部署资源要求高；运维依赖专业团队
- **性能参数**：
  - 响应延迟：P50 < 1ms, P99 1~5ms（准内存架构，金融级优化）
  - 读 QPS：数十万~百万（集群，TPC-C 基准测试中领先）
  - 写 QPS：数十万~80万（集群，Paxos 组内高性能写入）
  - 备注：延迟和吞吐在分布式数据库中断层领先（金融场景深度优化）

> **同类对比**：MySQL 是单机根基，PostgreSQL 是分析型关系库，TiDB/CockroachDB 解决弹性扩展，OceanBase 专注金融级。

## 二、文档型 NoSQL

| 数据库 | 水平扩展 | 一致性 | 读 QPS (单机) | 写 QPS (单机) | P99 延迟 | 特色 |
|--------|:--:|--------|:--:|:--:|:--:|------|
| MongoDB | ✅ 原生分片 | 最终/强可配 | 1万~5万 | 5千~2万 | 1~20ms | 生态最成熟 |
| DynamoDB | ✅ 全托管自动 | 最终/强可配 | 自动扩缩 | 自动扩缩 | <10ms (SLA) | AWS 原生 |
| Couchbase | ✅ 集群扩展 | 强一致 | 10万+ | 5万+ | <1ms | 缓存+文档融合 |
| CouchDB | ✅ 集群(复制) | 最终一致 | 1千~1万 | 1千~5千 | 5~50ms | 离线同步 |
| Firestore | ✅ 全托管自动 | 强/最终可配 | 自动扩缩 | 1万/s (限制) | <30ms | Google 原生 |

### 8. MongoDB
- **数据类型**：半结构化/非结构化（JSON/BSON 文档）
- **典型场景**：用户画像、CMS、IoT 数据、实时分析、目录数据
- **核心特点**：Schema-less 灵活模式、嵌套文档、原生水平分片、支持事务
- **缺点**：JOIN 能力弱；大量关联查询需应用层处理；内存消耗大
- **性能参数**：
  - 响应延迟：P50 < 1ms, P99 1~20ms（单文档 CRUD）
  - 读 QPS：1万~5万（单节点，索引命中，WiredTiger 引擎）
  - 写 QPS：5千~2万（单节点，默认 write concern: majority）
  - 备注：分片集群可线性扩展读写；write concern 越低写越快但一致性越弱

### 9. DynamoDB
- **数据类型**：键值/文档型（AWS 全托管）
- **典型场景**：Serverless 后端、IoT 设备数据、游戏排行榜、购物车、用户会话
- **核心特点**：全托管免运维、单毫秒延迟 SLA、TTL 自动过期、Streams 事件驱动
- **缺点**：AWS 锁定；费用随容量递增；查询能力弱于 MongoDB；二级索引有限
- **性能参数**：
  - 响应延迟：P50 < 5ms, P99 < 10ms（SLA 保证单毫秒延迟）
  - 读 QPS：自动扩缩（需预置 RCU/WCU），On-demand 模式可达数万+
  - 写 QPS：自动扩缩，单分区 1000 WCU（1KB = 1 WCU）
  - 备注：性能由配置的 RCU/WCU 决定；热分区问题需关注

### 10. Couchbase
- **数据类型**：JSON 文档 + 键值缓存一体化
- **典型场景**：移动 App 离线优先（Couchbase Lite）、实时游戏、用户 Profile
- **核心特点**：内存优先架构（Memcached 兼容）、N1QL 类 SQL 查询、多 DC 复制
- **缺点**：社区小生态弱；国内案例少；运维门槛较高
- **性能参数**：
  - 响应延迟：P50 < 1ms（内存命中）, P99 < 5ms
  - 读 QPS：10万+（单节点，内存架构，类 Memcached 性能）
  - 写 QPS：5万+（单节点，异步刷盘）
  - 备注：缓存+持久层融合，读延迟接近纯缓存；集群模式线性扩展

### 11. CouchDB
- **数据类型**：JSON 文档（RESTful）
- **典型场景**：移动/Web 离线同步、边缘数据采集、P2P 数据复制
- **核心特点**：HTTP API、MVCC、主主复制、CouchDB + PouchDB 实现客户端到服务端同步
- **缺点**：查询能力弱（需预定义视图）；不擅长高并发写入；生态工具少
- **性能参数**：
  - 响应延迟：P50 5~20ms, P99 20~50ms（视图查询需预热）
  - 读 QPS：1千~1万（单节点，视图索引后）
  - 写 QPS：1千~5千（单节点，MVCC 增加写开销）
  - 备注：写性能明显弱于 MongoDB；视图需要增量构建（首次慢）

### 12. Firestore
- **数据类型**：文档/集合（Google Cloud 全托管）
- **典型场景**：移动 App、实时协作应用、Firebase 生态项目
- **核心特点**：实时监听（Realtime Sync）、自动扩缩、多层级安全规则
- **缺点**：Google 锁定；复杂查询能力弱；费用模型复杂
- **性能参数**：
  - 响应延迟：P50 5~15ms, P99 < 30ms（全球部署 Edge 加速）
  - 读 QPS：自动扩缩（有默认配额限制，可申请提升）
  - 写 QPS：上限 ~1 万 writes/s（硬限制，需联系提升）
  - 备注：实时监听模式下客户端本地缓存可降低感知延迟

> **同类对比**：MongoDB 通用首选；DynamoDB AWS 原生免运维；Couchbase 移动端同步+高性能；CouchDB 边缘/离线场景；Firestore Google Firebase 生态。

## 三、宽列 NoSQL

| 数据库 | 水平扩展 | 一致性 | 读 QPS (单节点) | 写 QPS (单节点) | P99 延迟 | 适用规模 |
|--------|:--:|--------|:--:|:--:|:--:|:--:|
| Cassandra | ✅ 去中心化线性 | 最终/可调 | 1万~5万 | 5万~20万 | <10ms | 超大 |
| HBase | ✅ 依托 HDFS | 强一致 | 1万~5万 | 1万~5万 | 5~50ms | 超大 |
| ScyllaDB | ✅ 去中心化线性 | 最终/可调 | 5万~20万 | 20万~100万 | <5ms | 超大 |
| Bigtable | ✅ 全托管自动 | 强一致 | 自动扩缩 | 自动扩缩 | <10ms | 超大 |

### 13. Cassandra
- **数据类型**：大规模分布式写入密集型数据
- **典型场景**：IoT 时序事件、消息归档、推荐系统埋点、跨地域多活
- **核心特点**：无中心节点、线性扩展、天然多 DC 复制、写性能极好
- **缺点**：无 JOIN、无事务、无二级索引（3.0+ 有 MV）、CQL 限制多；磁盘空间放大
- **性能参数**：
  - 响应延迟：P50 < 1ms, P99 < 10ms（QUORUM 读写）
  - 读 QPS：1万~5万（单节点，SSD）
  - 写 QPS：5万~20万（单节点，LSM-Tree 顺序写入，极高吞吐）
  - 备注：写入通常 2~10 倍快于读取；三节点集群可达数十万写 QPS

### 14. HBase
- **数据类型**：稀疏宽表 / 海量结构化半结构化数据
- **典型场景**：日志存储、用户行为历史、元数据管理、时序存储
- **核心特点**：基于 HDFS 持久化、强一致性读写、RowKey 设计决定性能
- **缺点**：依赖 Hadoop 全家桶（HDFS/ZK）；运维重；写入吞吐低于 Cassandra
- **性能参数**：
  - 响应延迟：P50 1~5ms, P99 5~50ms（RowKey 点查）
  - 读 QPS：1万~5万（单 RegionServer，BlockCache 命中）
  - 写 QPS：1万~5万（单 RegionServer，先写 WAL 再写 MemStore）
  - 备注：RowKey 散列设计不当会导致热点 Region；大规模扫描延迟高

### 15. ScyllaDB
- **数据类型**：大规模分布式写入密集型（Cassandra 兼容）
- **典型场景**：Cassandra 替代方案、需要更低延迟更高吞吐的场景
- **核心特点**：C++ 重写 Cassandra、Seastar 异步框架、核绑定、延迟和吞吐大幅领先 Cassandra
- **缺点**：社区版功能受限；Java UDF 不兼容；运维工具链不如 Cassandra
- **性能参数**：
  - 响应延迟：P50 < 0.5ms, P99 < 5ms（官方宣称延迟为 Cassandra 的 1/5）
  - 读 QPS：5万~20万（单节点，核绑定架构）
  - 写 QPS：20万~100万+（单节点，异步无锁写入）
  - 备注：性能通常比 Cassandra 高 2~5 倍，延迟极低且更稳定

### 16. Bigtable
- **数据类型**：宽列（Google Cloud 全托管）
- **典型场景**：IoT 时序、用户行为日志、金融交易记录、广告技术（实时竞价）
- **核心特点**：HBase 协议兼容、全托管、低延迟、与 GCP 生态深度集成
- **缺点**：Google 锁定；费用随规模递增；开源替代缺乏
- **性能参数**：
  - 响应延迟：P50 < 5ms, P99 < 10ms（Google SLO 保证）
  - 读 QPS：自动扩缩（取决于节点数，单节点约 1 万 QPS）
  - 写 QPS：自动扩缩，单节点约 1 万 writes/s
  - 备注：全托管免配置性能；SSD 型号延迟更低但成本更高

> **同类对比**：Cassandra 开源首选；HBase Hadoop 生态配套；ScyllaDB 极致性能；Bigtable GCP 全托管。

## 四、键值存储

| 数据库 | 类型 | 水平扩展 | 读 QPS (单节点) | 写 QPS (单节点) | P99 延迟 | 核心用途 |
|--------|------|:--:|:--:|:--:|:--:|------|
| Redis | 内存 KV | ✅ 集群 | 10万~20万 | 8万~15万 | <1ms | 缓存+数据结构 |
| etcd | 分布式 KV | ✅ Raft | 1万~3万 | ~1万 (Raft) | 1~5ms | 配置/发现 |
| LevelDB | 嵌入式 LSM | ❌ | 1万~5万 | 5万~20万 | <1ms | 本地持久化引擎 |
| RocksDB | 嵌入式 LSM | ❌ | 5万~50万 | 20万~100万 | <1ms | 高性能引擎底层 |
| Memcached | 纯内存 | ✅ 分布式 | 20万~50万 | 20万~50万 | <0.5ms | 极简 KV 缓存 |
| Aerospike | 混合存储 | ✅ 集群 | 10万~25万 | 10万~25万 | <1ms | 低延迟广告/推荐 |
| FoundationDB | 分布式 KV | ✅ ACID | 5万~10万 | 5万~10万 | <1ms | 强一致基础层 |
| BoltDB | 嵌入式 B+Tree | ❌ | 5千~2万 | ~100 (串行写) | <1ms (读) | Go 嵌入式 KV |

### 17. Redis
- **数据类型**：热数据 / 缓存 / 多种数据结构
- **典型场景**：缓存（Session/热点）、分布式锁、排行榜（ZSet）、计数器/限流、消息队列（Stream）
- **核心特点**：极快纯内存读写、多种数据结构（String/Hash/List/Set/ZSet/Stream/Geospatial）、集群+持久化
- **缺点**：内存容量受限（成本高）；持久化有性能损耗；集群跨 slot 操作受限
- **性能参数**：
  - 响应延迟：P50 < 0.1ms, P99 < 1ms（单机，纯内存网络 IO）
  - 读 QPS：10万~20万（单机，Pipelining 可更高）
  - 写 QPS：8万~15万（单机，AOF 持久化会略降）；无持久化可达 15万+
  - 备注：内存数据库延迟天花板，主要受网络 RTT 和 CPU 瓶颈限制

### 18. etcd
- **数据类型**：配置 / 元数据 / 服务发现
- **典型场景**：K8s 核心存储、服务注册发现、分布式锁/选主、配置中心
- **核心特点**：强一致性（Raft）、Watch 机制、MVCC、轻量级
- **缺点**：存储容量有限（建议 < 8GB）；大量写入性能差；不适合业务数据
- **性能参数**：
  - 响应延迟：P50 1~3ms, P99 3~10ms（Raft 共识有额外开销）
  - 读 QPS：1万~3万（单节点，线性一致读）
  - 写 QPS：~1万（3 节点集群，Raft 串行化写入，受限于磁盘 fsync）
  - 备注：写瓶颈在 Raft 日志 fsync，使用 SSD 可达 ~1 万 writes/s

### 19. LevelDB
- **数据类型**：本地持久化键值（嵌入式）
- **典型场景**：Chrome IndexedDB 底层、移动端本地存储、嵌入式引擎
- **核心特点**：LSM-Tree 架构、写性能优秀、Google 开源
- **缺点**：单机单进程；多线程支持弱；随机读性能一般
- **性能参数**：
  - 响应延迟：P50 < 0.5ms (顺序), P99 < 10ms（随机读可能触发多层级查找）
  - 读 QPS：1万~5万（单机，SSD）
  - 写 QPS：5万~20万（单机，LSM-Tree 顺序写入）
  - 备注：写入远快于读取（LSM 特性）；数据量大时 Compaction 可能卡顿

### 20. RocksDB
- **数据类型**：本地持久化键值（嵌入式引擎）
- **典型场景**：数据库底层引擎（TiKV/MyRocks/ArangoDB）、Flink 状态后端
- **核心特点**：LevelDB 优化版，多线程 Compaction、布隆过滤器、列族、写放大控制
- **缺点**：配置项繁多（调优门槛高）；不具备分布式能力
- **性能参数**：
  - 响应延迟：P50 < 0.3ms, P99 < 5ms（布隆过滤 + 多线程Compaction）
  - 读 QPS：5万~50万（单机 SSD，布隆过滤器减少磁盘查找）
  - 写 QPS：20万~100万+（多线程 Compaction + 列族优化）
  - 备注：性能比 LevelDB 高 3~10 倍；Facebook 用其支撑亿级 QPS（多机）

### 21. Memcached
- **数据类型**：键值缓存 / 纯内存
- **典型场景**：数据库查询缓存、页面片段、Session 集中存储、API 限流
- **核心特点**：极简设计、多线程高并发、一致性哈希分布式
- **缺点**：无持久化、无复杂数据结构、数据类型简单
- **性能参数**：
  - 响应延迟：P50 < 0.1ms, P99 < 0.5ms（极简协议，纯内存）
  - 读 QPS：20万~50万+（单机，多线程无锁）
  - 写 QPS：20万~50万+（单机，与读性能接近）
  - 备注：比 Redis 更快（无数据结构开销），但功能较少

### 22. Aerospike
- **数据类型**：键值数据（SSD 优化混合存储）
- **典型场景**：广告平台实时竞价（RTB）、推荐引擎、风控系统、金融交易读写
- **核心特点**：SSD 优化、单毫秒延迟、混合内存+闪存架构、企业级高可用
- **缺点**：社区版受限；查询模型简单；国内生态弱
- **性能参数**：
  - 响应延迟：P50 < 0.5ms, P99 < 1ms（SSD 直写优化）
  - 读 QPS：10万~25万+（单节点，内存+SSD 混合命中）
  - 写 QPS：10万~25万+（单节点，闪存优化，写入与读性能接近）
  - 备注：SSD 性价比高，每 TB 成本远低于纯内存方案

### 23. FoundationDB
- **数据类型**：分布式 ACID 事务 KV（Apple 出品）
- **典型场景**：分布式元数据存储、强一致性业务状态、作为上层数据库引擎的底层存储
- **核心特点**：严格可串行化（Serializable）、确定性模拟器测试、分层架构
- **缺点**：社区小；SQL/文档层需配合其它工具；运维经验稀缺
- **性能参数**：
  - 响应延迟：P50 < 0.5ms, P99 < 5ms
  - 读 QPS：5万~10万+（集群，串行化读有一定开销）
  - 写 QPS：5万~10万+（集群，严格串行化事务写入）
  - 备注：ACID 事务有额外延迟开销；吞吐大幅领先传统关系型

### 24. BoltDB
- **数据类型**：Go 嵌入式键值（B+Tree）
- **典型场景**：Go 应用本地存储、etcd v2 底层引擎、CLI 工具本地数据库
- **核心特点**：纯 Go 实现、B+Tree 结构、单文件、ACID 事务、协程安全
- **缺点**：只读并发强但写串行；数据量大后性能下降；不可远程访问
- **性能参数**：
  - 响应延迟：P50 < 0.1ms（读）, P50 < 1ms（写）
  - 读 QPS：5千~2万（读事务可并发）
  - 写 QPS：~100~500（串行写锁，MMAP 页数决定）
  - 备注：写操作严格串行，高并发写场景不适合；适合读多写少的 Go 应用

> **同类对比**：Redis 通用缓存王者；etcd 分布式协调标准；RocksDB 引擎之王；Memcached 极简高性能；Aerospike 广告推荐专用；BoltDB Go 嵌入式首选。

## 五、时序数据库 (TSDB)

| 数据库 | 水平扩展 | SQL 支持 | 写入吞吐 (单节点) | 查询延迟 | 存储引擎 | 特色 |
|--------|:--:|:--:|:--:|:--:|--------|------|
| InfluxDB | ✅ 集群版(企业) | 类 SQL | 50万~100万 points/s | 1~100ms | TSM | 专注时序 |
| TimescaleDB | ✅ PG 扩展 | 完整 SQL | 10万~100万 metrics/s | 1~100ms | Hypertable | 时序+关系融合 |
| Prometheus | ✅ Federation | PromQL | 10万~50万 samples/s | <100ms | 本地 TSDB | 监控专精 |
| TDengine | ✅ 原生集群 | 类 SQL | 300万~600万 points/s | <10ms | 自研引擎 | IoT 优等生 |
| VictoriaMetrics | ✅ 集群 | PromQL 兼容 | 100万~300万 samples/s | <50ms | 自研 | 长周期监控存储 |

### 25. InfluxDB
- **数据类型**：时间序列指标/监控
- **典型场景**：服务器监控、IoT 传感器、金融行情、DevOps 指标
- **核心特点**：高写入吞吐、自动降采样和过期策略、类 SQL 查询
- **缺点**：集群版收费；开源版无分布式；生态不如 Prometheus 在监控领域
- **性能参数**：
  - 响应延迟：P50 1~10ms（简单查询）, P99 10~100ms（复杂聚合）
  - 写入吞吐：50万~100万 points/s（单节点，TSM 引擎优化）
  - 查询 QPS：数百~数千（聚合查询为主，取决于时间范围和精度）
  - 备注：写入吞吐极高；复杂聚合查询（高精度长范围）延迟较大

### 26. TimescaleDB
- **数据类型**：时序 + 关系型结合
- **典型场景**：物联网分析、金融交易记录、DevOps 监控（嵌入式 PG 生态）
- **核心特点**：完整 SQL + JOIN、自动分区（hypertable）、高压缩比、PG 生态兼容
- **缺点**：写入吞吐不如 InfluxDB；依赖 PG 扩展限制
- **性能参数**：
  - 响应延迟：P50 1~10ms, P99 10~100ms（完整 SQL 分析）
  - 写入吞吐：10万~100万 metrics/s（单节点，受 PG 写入限制）
  - 查询 QPS：数百~数千（JOIN 查询取决于复杂度）
  - 备注：写入不如专用 TSDB，但查询灵活性（SQL+JOIN）是独特优势

### 27. Prometheus
- **数据类型**：监控指标时序（拉模式）
- **典型场景**：K8s 集群监控、微服务指标采集、告警、Grafana 数据源
- **核心特点**：Pull 拉取模式、多维标签、PromQL、服务发现、内置告警
- **缺点**：不适合长期存储（数据有保质期）；单点写入；查询分析能力弱于 InfluxDB
- **性能参数**：
  - 响应延迟：P50 < 5ms, P99 < 100ms（复杂 PromQL 聚合）
  - 写入吞吐：10万~50万 samples/s（单节点，本地 TSDB，受 scrape 间隔影响）
  - 查询 QPS：数百（单机，PromQL 即时查询+范围查询）
  - 备注：面向每秒数百万 active series 设计；横向扩展依赖 Federation/Thanos

### 28. TDengine
- **数据类型**：IoT 时序数据（国产开源）
- **典型场景**：物联网/车联网传感器、工业互联网、智慧城市数据采集
- **核心特点**：超级表模型、列式存储、10 倍以上压缩比、极简运维、国产开源
- **缺点**：SQL 兼容性有限；生态工具不如 InfluxDB；非 IoT 场景适用性一般
- **性能参数**：
  - 响应延迟：P50 < 1ms, P99 < 10ms（超级表聚合查询）
  - 写入吞吐：300万~600万+ points/s（单节点，官方基准测试）
  - 查询 QPS：数千~数万（列存 + 超级表预聚合）
  - 备注：IoT 场景写入吞吐遥遥领先；压缩比 10~20 倍减少存储成本

### 29. VictoriaMetrics
- **数据类型**：Prometheus 兼容长周期监控数据
- **典型场景**：大规模 K8s/Prometheus 长期存储、替代 Thanos/Cortex
- **核心特点**：PromQL/Prometheus 全兼容、单二进制部署、压缩比极高、内存占用低
- **缺点**：社区版功能有裁剪；不适合非监控场景
- **性能参数**：
  - 响应延迟：P50 < 1ms, P99 < 50ms（PromQL 兼容查询）
  - 写入吞吐：100万~300万 samples/s（单节点，压缩写入）
  - 查询 QPS：数千（单节点，远超原生 Prometheus）
  - 备注：比 Prometheus 内存少 7 倍，磁盘少 5 倍，性能高2~5倍

> **同类对比**：监控场景 Prometheus + VictoriaMetrics 长存；时序分析+SQL TimescaleDB；高吞吐时序 InfluxDB；IoT 场景 TDengine。

## 六、列式 OLAP / 分析型

| 数据库 | 实时性 | 查询延迟 | 读 QPS (集群) | 写入吞吐 | 水平扩展 | 核心场景 |
|--------|:--:|:--:|:--:|:--:|:--:|------|
| ClickHouse | 近实时 | 10ms~1s | 数百~数千 | 100万~500万 rows/s | ✅ | 实时多维分析 |
| StarRocks/Doris | 实时 | 10~100ms | 数千~数万 | 50万~200万 rows/s | ✅ | 高并发实时分析 |
| Hive | 离线 (T+1) | 分钟~小时 | — | 批处理 | ✅ | 离线批处理 |
| Druid | 实时 | 亚秒 (<100ms) | 数百~数千 | 10万~100万 events/s | ✅ | 实时摄入+分析 |
| Kylin | 准实时 | 亚秒 (<100ms) | 数千~数万 | Cube 构建(小时级) | ✅ | 预计算 Cube |
| Presto/Trino | 准实时 | 1s~10s | 数十~数百 | 取决于源 | ✅ | 跨源联邦查询 |

### 30. ClickHouse
- **数据类型**：海量结构化数据的极速分析
- **典型场景**：实时报表、用户行为分析、广告统计、日志分析、BI 看板
- **核心特点**：列式存储 + 向量化计算、查询极快、压缩比高、写入吞吐大
- **缺点**：频繁 UPDATE/DELETE 极差；JOIN 能力弱于 MPP 数据库
- **性能参数**：
  - 响应延迟：P50 10~100ms, P99 100ms~1s（大表聚合）
  - 读 QPS：数百~数千（集群，分析查询通常不追求高并发）
  - 写入吞吐：100万~500万 rows/s（单节点，MergeTree 批量写）
  - 备注：单表亿级数据聚合亚秒级；不擅长 UPDATE/DELETE 和点查

### 31. StarRocks / Doris
- **数据类型**：实时 OLAP 多维分析
- **典型场景**：实时看板、用户画像标签、多表关联分析、HTAP 混合场景
- **核心特点**：MPP 架构、向量化引擎、标准 SQL + 物化视图、高并发低延迟
- **缺点**：生态不如 ClickHouse 成熟；单表聚合性能略逊 ClickHouse
- **性能参数**：
  - 响应延迟：P50 10~50ms, P99 50~100ms（多表 JOIN，MPP 并行）
  - 读 QPS：数千~数万（高并发场景优于 ClickHouse）
  - 写入吞吐：50万~200万 rows/s（集群，支持实时微批导入）
  - 备注：多表 JOIN 高并发场景领先 ClickHouse；适合面向用户的实时看板

### 32. Hive
- **数据类型**：大规模离线批处理（PB 级）
- **典型场景**：离线 ETL、数仓分层（ODS/DWD/DWS/ADS）、历史归档、T+1 报表
- **核心特点**：SQL-on-Hadoop、HQL 语法、UDF 丰富、低成本海量存储
- **缺点**：延迟极高（分钟级以上）；不适合实时分析；架构笨重
- **性能参数**：
  - 响应延迟：P50 数分钟, P99 数小时（大规模批处理）
  - 查询并发：极低（2~10），离线场景不追求并发
  - 数据吞吐：TB~PB 级/天（批处理，非实时 QPS）
  - 备注：不适用 QPS 概念；以小时至天为周期的批量计算引擎

### 33. Apache Druid
- **数据类型**：实时摄入 + 多维分析
- **典型场景**：实时广告效果分析、实时用户行为看板、网络流量分析、APM 监控
- **核心特点**：实时 Kafka 摄入、预聚合+索引加速、亚秒级查询、自动数据分区和过期
- **缺点**：数据变更困难（不擅长 UPDATE）；部署组件多运维重；SQL 兼容不如 StarRocks
- **性能参数**：
  - 响应延迟：P50 < 10ms, P99 < 100ms（预聚合 + 位图索引加速）
  - 读 QPS：数百~数千（集群，OLAP 聚合查询）
  - 写入吞吐：10万~100万 events/s（集群，实时 Kafka 摄入）
  - 备注：亚秒级延迟是核心卖点；预聚合在内存+SSD，牺牲灵活性换取速度

### 34. Apache Kylin
- **数据类型**：Cube 预计算 OLAP
- **典型场景**：大数据量固定维度分析、企业 BI 报表（Tableau/PowerBI 对接）、Hadoop 生态分析
- **核心特点**：Cube 预加速、亚秒级查询、标准 SQL/JDBC/ODBC、Hadoop 原生集成
- **缺点**：必须预设维度和指标（Cube 设计）；Cube 构建耗时；不适合灵活探索分析
- **性能参数**：
  - 响应延迟：P50 < 10ms, P99 < 100ms（Cube 命中的预计算查询）
  - 读 QPS：数千~数万（Cube 查询类似 KV 查找，高并发能力强）
  - Cube 构建：小时级（百亿行数据），增量分钟级
  - 备注：查询极快但灵活性差；维度变更需重建 Cube；适合固定报表

### 35. Presto / Trino
- **数据类型**：联邦查询引擎（非数据库）
- **典型场景**：跨 MySQL/Hive/MongoDB/Kafka 等多个数据源的统一 SQL 查询
- **核心特点**：内存计算、不持久化存储、多数据源联邦 JOIN、标准 ANSI SQL
- **缺点**：无自身存储；对大表聚合不如 ClickHouse；需要底层源稳定
- **性能参数**：
  - 响应延迟：P50 1s~5s, P99 10s~60s（取决于源数据和 JOIN 复杂度）
  - 查询并发：数十~数百（内存计算，依赖 Worker 节点数）
  - 数据吞吐：取决于底层数据源
  - 备注：没有自身存储和索引，查询速度依赖于数据源；不适合高并发

> **同类对比**：极速单表聚合 ClickHouse；多表 JOIN 高并发 StarRocks/Doris；离线批处理 Hive；实时摄入 Druid；固定报表 Kylin；跨源查询 Presto/Trino。

## 七、搜索引擎

| 数据库 | 水平扩展 | 近实时 | 读 QPS (单节点) | 写 QPS (单节点) | P99 延迟 | 核心场景 |
|--------|:--:|:--:|:--:|:--:|:--:|------|
| Elasticsearch | ✅ | ✅ (1s) | 数千~数万 | 1万~5万 qps | 50~500ms | 搜索+日志 |
| Solr | ✅ | ✅ (秒级) | 数千~数万 | 1万~3万 qps | 50~500ms | 搜索平台 |
| MeiliSearch | ⚠️ 有限 | ✅ (ms) | 数百~数千 | ~100 qps | <10ms | 轻量站内搜索 |

### 36. Elasticsearch
- **数据类型**：全文检索 + 日志/时序
- **典型场景**：全文搜索、ELK 日志分析、监控告警、商品/站内搜索
- **核心特点**：倒排索引、近实时搜索、聚合分析、作为搜索层而非主数据库
- **缺点**：不适合做主库；大量写入查询下降；集群管理需谨慎
- **性能参数**：
  - 响应延迟：P50 10~50ms, P99 50~500ms（全文搜索 + 聚合分析）
  - 读 QPS：数千~数万（集群，取决于索引规模和查询复杂度）
  - 写 QPS：1万~5万 qps（单节点），批量写入可更高
  - 备注：写入后 1s 内可搜索（refresh interval）；深度分页性能差

### 37. Apache Solr
- **数据类型**：全文检索 + 结构化搜索
- **典型场景**：企业内网搜索、电商分类导航（Faceted Search）、知识库检索
- **核心特点**：比 ES 更专注搜索精确性（而非日志分析）、Facet 强大、XML/JSON/CSV 多格式索引
- **缺点**：部署比 ES 复杂；社区不如 ES 活跃；实时性弱于 ES
- **性能参数**：
  - 响应延迟：P50 10~50ms, P99 50~500ms
  - 读 QPS：数千~数万（集群，Facet 搜索场景）
  - 写 QPS：1万~3万 qps（单节点）
  - 备注：精确搜索准确率高于 ES；实时性弱于 ES（近实时秒级）

### 38. MeiliSearch
- **数据类型**：全文检索（轻量级）
- **典型场景**：小型网站站内搜索、文档站内搜索、MVP 原型搜索
- **核心特点**：超轻量（单二进制）、毫秒级搜索、中文分词开箱即用、API 极简
- **缺点**：大规模扩展能力弱、功能远不如 ES/Solr 丰富
- **性能参数**：
  - 响应延迟：P50 < 5ms, P99 < 10ms（轻量级倒排索引，毫秒响应）
  - 读 QPS：数百~数千（单节点）
  - 写 QPS：~100 qps（单节点，非批量场景）
  - 备注：延迟极低但吞吐有限；百万级文档以下性能优秀

> **同类对比**：大型搜索+日志选 ES；企业精确搜索选 Solr；小站/原型选 MeiliSearch。

### 39. Neo4j
- **数据类型**：高度关联/网络化数据
- **典型场景**：社交关系网络、推荐系统、知识图谱、欺诈检测、供应链追溯
- **核心特点**：原生图存储计算、Cypher 查询、N 度人脉深度关系
- **缺点**：写扩展不如分布式图库；超大图内存占用高；企业版收费
- **性能参数**：
  - 响应延迟：P50 1~10ms（2 度关系）, P99 10~100ms（4~6 深度遍历）
  - 读 QPS：1千~1万（单节点，取决于图深度和广度）
  - 写 QPS：数百~数千（单节点，节点+边创建）
  - 备注：深度遍历是核心优势（RDBMS 几十秒，Neo4j 毫秒级）；超十亿节点需集群

### 40. NebulaGraph
- **数据类型**：大规模分布式图数据（国产开源）
- **典型场景**：亿级顶点知识图谱、社交网络、金融风控关联关系、推荐召回
- **核心特点**：存算分离、Shared-Nothing 分布式、水平扩展、支持千亿级边
- **缺点**：社区和学习资源少；nGQL 学习成本；运维组件多（Meta/Graph/Storage）
- **性能参数**：
  - 响应延迟：P50 1~5ms, P99 5~50ms（6 度遍历，分布式并行）
  - 读 QPS：数万~数十万（集群，Shared-Nothing 并行）
  - 写 QPS：数万~数十万（集群，分布式写入）
  - 备注：千亿级边规模下仍保持毫秒延迟；分布式大图场景唯一选择

### 41. Dgraph
- **数据类型**：分布式图数据（GraphQL 原生）
- **典型场景**：GraphQL API 后端、社交平台关注关系、权限 RBAC 图
- **核心特点**：GraphQL± 查询、原生分布式、ACID 事务、Go 实现
- **缺点**：社区规模小；被 Hypermode 收购后发展方向不明；国内生态弱
- **性能参数**：
  - 响应延迟：P50 1~10ms, P99 10~100ms（GraphQL 查询）
  - 读 QPS：数千~数万（集群）
  - 写 QPS：数千~数万（集群，分布式事务写入）
  - 备注：GraphQL 原生使 API 层零代码；写吞吐略弱于 NebulaGraph

> **同类对比**：中小图 Neo4j；超大规模 NebulaGraph；GraphQL 原生 Dgraph。

## 九、消息/流平台

| 平台 | 水平扩展 | 持久化 | 单 Broker 吞吐 | 端到端延迟 | 协议 | 核心定位 |
|------|:--:|:--:|:--:|:--:|--------|------|
| Kafka | ✅ | ✅ | 100万~200万 msgs/s | 1~10ms | 自有 | 高吞吐流 |
| RabbitMQ | ✅ | ✅ | 1万~5万 msgs/s | <1ms | AMQP | 可靠消息 |
| Pulsar | ✅ | ✅+分层 | 100万~200万 msgs/s | 1~5ms | 自有 | 云原生流 |
| RocketMQ | ✅ | ✅+事务 | 10万~100万 msgs/s | <1ms | 自有 | 分布式事务消息 |
| NATS | ✅ | ✅ (JetStream) | 1000万+ msgs/s | <0.1ms | 自有 | 轻量云原生 |

### 42. Kafka
- **数据类型**：流式数据 / 事件
- **典型场景**：消息队列、日志采集、数据管道、事件溯源、埋点追踪、流计算
- **核心特点**：极高吞吐（百万条/秒）、持久化+回溯、发布订阅
- **缺点**：运维复杂；消息延迟较高（批处理优化）；无内置路由
- **性能参数**：
  - 延迟：P50 1~3ms, P99 5~10ms（Producer → Broker → Consumer）
  - 吞吐：100万~200万 msgs/s（单 Broker），500万~1000万+（集群）
  - 备注：吞吐极高但延迟不如 RabbitMQ/NATS；批处理模式延迟敏感场景需关闭 batching

### 43. RabbitMQ
- **数据类型**：异步消息 / 任务分发
- **典型场景**：任务队列、应用解耦、流量削峰、RPC、订单异步处理
- **核心特点**：AMQP 标准、灵活路由（Direct/Topic/Fanout）、消息确认+死信
- **缺点**：吞吐低于 Kafka；大量堆积性能下降；持久化模式慢
- **性能参数**：
  - 延迟：P50 < 0.5ms, P99 < 5ms（非持久化模式）
  - 吞吐：1万~5万 msgs/s（单 Broker，持久化 + ACK）；非持久化可达 5万~10万
  - 备注：单消息延迟极低但吞吐有限；持久化+HA 模式吞吐大幅下降

### 44. Pulsar
- **数据类型**：流式 / 事件 / 消息
- **典型场景**：多租户事件中心、跨地域复制、IoT 消息、计费通知
- **核心特点**：计算存储分离、原生多租户、分层存储（S3/HDFS）、低延迟
- **缺点**：运维复杂（BookKeeper+ZK）；社区小于 Kafka
- **性能参数**：
  - 延迟：P50 1~3ms, P99 < 5ms（BookKeeper 持久化写入）
  - 吞吐：100万~200万 msgs/s（单 Broker），计算与存储可独立扩展
  - 备注：计算存储分离使扩展更灵活；分层存储降低冷数据成本

### 45. RocketMQ
- **数据类型**：事务消息 / 异步消息（阿里开源）
- **典型场景**：分布式事务最终一致性（交易/库存）、电商异步解耦、金融通知
- **核心特点**：分布式事务消息、顺序消息、延迟消息、百万级吞吐、阿里背书+国内生态好
- **缺点**：依赖 NameServer 单点（5.0 已改进）；海外社区弱；部分功能商业化
- **性能参数**：
  - 延迟：P50 < 0.5ms, P99 < 5ms（同步发送）
  - 吞吐：10万~100万 msgs/s（单 Broker，取决于消息大小）
  - 备注：事务消息是核心差异（分布式事务场景首选）；国内大厂广泛验证

### 46. NATS
- **数据类型**：轻量级事件/消息
- **典型场景**：微服务间异步通信、边缘/IoT 设备消息、云原生 Service Mesh 控制面
- **核心特点**：极轻量（单一 Go 二进制）、千万级吞吐、低延迟、JetStream 持久化
- **缺点**：持久化能力弱于 Kafka/Pulsar；生态工具不如大型 MQ 丰富
- **性能参数**：
  - 延迟：P50 < 0.1ms, P99 < 0.5ms（at-most-once 非持久化）
  - 吞吐：1000万+ msgs/s（单节点，简单 pub/sub）
  - 备注：延迟和吞吐均为消息队列天花板；JetStream 持久化会降低到百万级

> **同类对比**：高吞吐流 Kakfa；业务可靠 RabbitMQ；分布式事务 RocketMQ；多租户 Pulsar；轻量云原生 NATS。

## 十、对象存储

| 存储 | 水平扩展 | 单请求延迟 | 读吞吐 | 写吞吐 | 一致性 | 核心场景 |
|------|:--:|:--:|:--:|:--:|--------|------|
| MinIO / S3 | ✅ 无限 | 10~100ms | GB/s 级 | GB/s 级 | 强（新版） | 对象/数据湖 |
| Ceph | ✅ 无限 | 1~30ms (SSD) | GB/s 级 | GB/s 级 | 强 | 统一存储 |
| HDFS | ✅ 线性 | 10~100ms | GB/s 级 | GB/s 级 | 强 | Hadoop 生态 |

### 47. MinIO / S3
- **数据类型**：非结构化大文件 / 二进制
- **典型场景**：图片/视频/文档存储（CDN 回源）、数据湖（Parquet/ORC/Iceberg）、备份归档、AI 训练
- **核心特点**：S3 兼容 API、无限扩展、静态加密、版本控制、生命周期管理
- **缺点**：不适合频繁更新小数据；延迟高于块存储；MinIO 社区版功能受限
- **性能参数**：
  - 响应延迟：P50 10~30ms, P99 50~100ms（取决于对象大小和网络）
  - 读吞吐：GB/s 级（集群，大文件顺序读）
  - 写吞吐：GB/s 级（集群，MinIO 单节点 1~5 GB/s）
  - 备注：对象存储延迟较高，不适合低延迟小 IO；面向吞吐而非 IOPS

### 48. Ceph
- **数据类型**：统一存储（对象/块/文件）
- **典型场景**：私有云存储底座（OpenStack）、虚拟机磁盘镜像、大规模冷热数据分层
- **核心特点**：CRUSH 算法去中心化、S3/Swift/Block/FS 四合一、自愈、弹性扩展
- **缺点**：运维极重（架构复杂）；性能调优门槛高；中小规模不如 MinIO 轻量
- **性能参数**：
  - 响应延迟：P50 1~5ms (块存储 SSD), P50 10~30ms (对象存储)
  - 块存储 IOPS：数万~数十万（集群，NVMe SSD）
  - 对象吞吐：GB/s 级（集群，RADOS Gateway）
  - 备注：块存储延迟远低于对象接口；统一存储架构带来运维复杂性

### 49. HDFS
- **数据类型**：Hadoop 生态文件系统
- **典型场景**：Hadoop/Spark/Hive 底层存储、大规模离线数据存算分离
- **核心特点**：成本低、海量数据顺序读写优秀、多副本容错、与 Hadoop 生态深度绑定
- **缺点**：不适合小文件（NameNode 压力）、不擅长随机读写、延迟高、不适合实时场景
- **性能参数**：
  - 响应延迟：P50 10~50ms, P99 100ms~1s（受 NameNode 查询和网络影响）
  - 顺序读吞吐：100~500 MB/s（单 DataNode，可线性扩展）
  - 顺序写吞吐：100~500 MB/s（单 DataNode）
  - 备注：不适合小文件 IO（NameNode 内存压力）；不适合低延迟随机访问

> **同类对比**：MinIO 轻量对象存储；Ceph 统一存储（块+对象+文件）；HDFS Hadoop 标配。

## 十一、向量数据库 (Vector DB) 🆕

> **新一代 AI 基础设施**：用于大模型 RAG、语义搜索、图像检索、推荐系统的向量相似度查询。

| 数据库 | 水平扩展 | 索引算法 | 查询延迟 (Top100) | QPS (单节点) | 维度上限 | 部署方式 |
|--------|:--:|--------|:--:|:--:|:--:|--------|
| Milvus | ✅ 分布式 | IVF/HNSW/DiskANN | 1~10ms | 1千~1万+ | 32768 | 自托管/云 |
| Weaviate | ✅ 集群 | HNSW/Flat | 1~10ms | 数百~数千 | 65536 | 自托管/云 |
| Pinecone | ✅ 云原生 | 自研 | <10ms | 自动扩缩 | 20000+ | SaaS |
| Qdrant | ✅ 集群 | HNSW | 1~5ms | 数千~数万 | 65536 | 自托管/云 |
| Chroma | ⚠️ 有限 | HNSW | 1~10ms | 数十~数百 | 20000 | 嵌入式 |

### 50. Milvus
- **数据特点**：高维向量嵌入（Embeddings）+ 结构化元数据
- **典型场景**：大模型 RAG 知识库、图像相似搜索、推荐系统召回、生物特征检索
- **核心特点**：存算分离、多种索引算法(HNSW/IVF/DiskANN)、支持标量+向量混合过滤、社区最活跃的向量库
- **缺点**：部署组件多；资源消耗大；小数据量有些重
- **性能参数**：
  - 查询延迟：P50 1~5ms, P99 < 10ms（HNSW, 百万级向量, Top100）
  - QPS：1千~1万+（单节点，HNSW），集群可线性扩展
  - 索引构建：~1 分钟/百万向量（HNSW）
  - 备注：支持 DiskANN 可将向量卸载到 SSD 降低成本

### 51. Weaviate
- **数据特点**：向量 + 传统数据模型混合（图+文档）
- **典型场景**：多模态搜索、知识图谱+语义搜索融合、可解释 RAG
- **核心特点**：原生 GraphQL API、内置向量化模块（自动生成 Embedding）、支持对象和向量混合索引
- **缺点**：纯大规模向量检索性能不如 Milvus；中文生态弱
- **性能参数**：
  - 查询延迟：P50 1~10ms, P99 10~50ms（混合图+向量查询）
  - QPS：数百~数千（单节点，混合索引开销略高）
  - 索引构建：~1 分钟/百万向量
  - 备注：混合检索（全文+向量+图）是独特优势

### 52. Pinecone
- **数据特点**：纯向量（SaaS）
- **典型场景**：快速搭建 RAG、不需要运维向量的初创团队
- **核心特点**：全托管免运维、自动索引优化、毫秒级查询、Serverless
- **缺点**：费用高；数据在云端（合规限制）；无法自定义部署
- **性能参数**：
  - 查询延迟：P50 < 5ms, P99 < 10ms（SaaS，自动索引优化）
  - QPS：自动扩缩（起步数百 QPS，付费越高上限越高）
  - 索引构建：自动（后台增量构建）
  - 备注：性能取决于付费等级；无需关心索引算法和参数调优

### 53. Qdrant
- **数据特点**：高性能向量（Rust 实现）
- **典型场景**：对延迟敏感的实时推荐、边缘设备向量检索
- **核心特点**：Rust 原生性能、内存友好、支持量化、极低延迟、REST/gRPC API
- **缺点**：社区小于 Milvus；生态工具少；企业版收费
- **性能参数**：
  - 查询延迟：P50 < 1ms, P99 < 5ms（Rust 原生，HNSW 索引）
  - QPS：数千~数万（单节点，Rust 无 GC，内存效率高）
  - 索引构建：~30 秒/百万向量
  - 备注：纯向量检索延迟最低的选项之一；量化降低内存 3~4 倍

### 54. Chroma
- **数据特点**：嵌入式轻量向量
- **典型场景**：LLM 原型开发、本地 RAG 实验、Jupyter Notebook 集成
- **核心特点**：pip install 即用、Python 友好、类似 SQLite 的零配置、适合开发阶段
- **缺点**：生产级扩展性弱；不适合大规模数据；分布式能力有限
- **性能参数**：
  - 查询延迟：P50 1~10ms, P99 < 50ms（本地 HNSW）
  - QPS：数十~数百（嵌入式，单进程内存模式）
  - 索引构建：~1 分钟/百万向量
  - 备注：开发/原型场景专用，不适合生产级高并发

> **同类对比**：企业级 Milvus；多模态 Weaviate；免运维 Pinecone；高性能 Qdrant；开发原型 Chroma。

## 十二、数据湖 / 湖仓格式 🆕

> **数据湖格式**：非数据库，是管理数据湖中文件（Parquet/ORC）的元数据层，提供 ACID、时间旅行、Schema 演化等能力。

| 格式 | 水平扩展 | ACID | 查询延迟 | 写入吞吐 | 生态 | 核心特色 |
|------|:--:|:--:|:--:|:--:|--------|------|
| Apache Iceberg | ✅ | ✅ | 秒~分钟 | 取决于引擎 | Spark/Flink/Trino/Presto | 开放标准、分区演化 |
| Delta Lake | ✅ | ✅ | 秒~分钟 | 取决于引擎 | Spark 生态 | Databricks 驱动 |
| Apache Hudi | ✅ | ✅ | 秒~分钟 | 取决于引擎 | Spark/Flink/Hive | 流式 Upsert 强项 |

### 55. Apache Iceberg
- **数据模型**：表格式（Table Format），管理 Parquet/ORC/Avro 数据文件
- **典型场景**：跨引擎统一数据湖（Spark/Flink/Trino 同一份数据）、慢速变化维度表（SCD）、合规审计（时间旅行）
- **核心特点**：真正开放标准（Apache 顶级项目）、分区演化无需重写数据、隐藏分区、快照隔离
- **缺点**：需要额外元数据服务（Catalog）；实时 Upsert 弱于 Hudi
- **性能参数**：
  - 查询延迟：秒~分钟（取决于底层引擎，如 Spark/Trino）
  - 写入吞吐：取决于底层计算引擎（Spark/Flink 的并行写入能力）
  - 元数据操作：分区演化对查询零开销；快照操作毫秒级
  - 备注：性能取决于底层计算+存储引擎，格式本身是元数据层

### 56. Delta Lake
- **数据模型**：表格式（Databricks/Linux Foundation）
- **典型场景**：Databricks 生态首选、Lakehouse 架构、ML 训练 pipeline
- **核心特点**：ACID 事务、时间旅行、Schema 强制+演化、Z-Ordering、DataFrame API 原生集成
- **缺点**：依赖 Spark 较重；与 Databricks 绑定趋势；引擎生态不如 Iceberg 广泛
- **性能参数**：
  - 查询延迟：秒~分钟（Spark 引擎优化，Photon 加速后更快）
  - 写入吞吐：Z-Ordering 优化的文件布局可提升查询 10~100 倍
  - 备注：Databricks 专属优化（Photon/Delta Engine）时性能最佳

### 57. Apache Hudi
- **数据模型**：表格式，专注于流式增量处理
- **典型场景**：近实时数仓、流式 Upsert/Delete、CDC 同步到数据湖、增量 ETL
- **核心特点**：增量 Upsert 强项、增量查询（Incremental Query）、MoR/CoW 双模式、自动文件大小控制
- **缺点**：并发查询性能弱于 Iceberg；文件管理更复杂；概念多学习成本高
- **性能参数**：
  - 写入延迟：分钟级（MoR 模式）, 分钟~小时级（CoW 模式）
  - 增量查询：秒~分钟（仅处理增量文件，效率高）
  - 备注：流式 Upsert 性能在三大格式中最优；CDC 场景首选

> **同类对比**：跨引擎统一选 Iceberg；Databricks/Spark 选 Delta Lake；流式 Upsert 选 Hudi。
